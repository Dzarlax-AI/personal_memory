Product-led README implementation plan

Prepared 2026-07-19 for the codex/service-hardening worktree.

Goal: replace the current reference-heavy README with professional English documentation for a universal, self-hosted MCP memory product. Users come first; developer and operator material follows.

Planning rule: this HTML file is the only planning artifact for the README rewrite. No separate specification will be generated.

1. Scope and success criteria

The work changes README.md only, plus this plan's completion record after implementation. It does not change application behavior, Compose topology, production infrastructure, licensing, or deployment state.

The rewrite is complete when a new reader can answer, in order:

  1. What problem does Personal Memory solve?
  2. Which capabilities are core and which are optional?
  3. What infrastructure is actually required?
  4. How do I deploy it, connect an MCP client, and use the first tools?
  5. How do I develop, secure, back up, migrate, and operate it?

2. Current behavior and verified gaps

  • The README leads with implementation components before clearly explaining the product's purpose and user outcomes.
  • It mixes user onboarding, architecture, operator runbooks, tool reference, and personal workflow guidance without a layered reading path.
  • It contains a duplicated Backups heading and personal conventions that should not define a universal project.
  • Architecture text says two services and external infra-qdrant; repository Compose currently defines three services and local memory-qdrant.
  • The tool list omits get_operational_context, exact point_id mutation paths, input bounds, and the authenticated plain-text operational endpoint.
  • RAG documentation omits generation-aware replacement and does not state that the provided Compose file needs an added documents bind mount.
  • The repository has no LICENSE or CONTRIBUTING file, so the README must not claim an open-source license or invent a contribution policy.
  • Named-client compatibility claims are temporally fragile and not fully test-backed by this repository.

4. Content decisions

  • Position the project as a universal self-hosted product, but avoid the legal phrase “open source” until a license exists. Use “self-hosted project” and “public source repository” where needed.
  • Describe persistent semantic facts as the core product. Mark RAG, Todoist, visualization, and OAuth as optional feature gates.
  • State accurately that fact/document embeddings remain inside the deployment, while Todoist and an OAuth issuer may involve external services.
  • Use generic placeholders such as example.com; remove Resilio-specific and personal namespace prescriptions from the main onboarding path.
  • Present repository Compose as a production-oriented baseline that expects an external traefik network and host bind directories. Do not call it a standalone one-command stack.
  • Call out the mismatch between production pinning guidance and the checked-in :latest application image without pretending it is fixed.
  • Use named client examples only where repository artifacts support them. Frame client-specific commands as examples that may need verification against current client documentation.
  • Keep safety-sensitive operational details: fail-closed auth, request limits, RAG generation replacement, snapshot restore, and exclusive maintenance-mode ID migration.

6. Sequential implementation tasks

Task 1 — Establish the product narrative

  1. Replace the title and opening paragraphs with a concise value proposition centered on durable, user-controlled context.
  2. Add “Why Personal Memory” and a four-capability overview.
  3. State privacy boundaries without absolute security or compliance claims.

Task 2 — Build the user journey

  1. Add a compact conceptual architecture diagram.
  2. Write an accurate quick start covering required external network, host directories, .env, Compose startup, logs, and /health.
  3. Explain the missing RAG documents mount when RAG is enabled.
  4. Provide client connection examples and intent-led usage examples for storing, recalling, organizing, document search, and optional Todoist.

Task 3 — Rebuild the reference layer

  1. Document all 12 memory tools, 2 gated RAG tools, and 7 gated Todoist tools with exact arguments.
  2. Document input limits, mutation confidence/ambiguity behavior, point_id, dry-run deletion, and conditional routes.
  3. Group environment variables by server/auth, memory, integrations, RAG, visualization, and backup instead of one undifferentiated table.

Task 4 — Add the developer/operator path

  1. Document the actual three-service Compose topology and current trust boundaries.
  2. Describe make test, raw Go checks, Docker build, shipped binaries, and repository layout.
  3. Preserve accurate RAG indexing, snapshot restore, safe ID migration, image pinning, deployment verification, and rollback guidance.
  4. Explicitly state that no license or contribution policy is currently provided instead of inventing one.

Task 5 — Edit for professional quality

  1. Remove duplicate headings, redundant explanations, personal namespace mandates, and unsupported named-client claims.
  2. Use consistent heading depth, blank-line spacing, tables, callouts, code fences, terminology, and US-English spelling.
  3. Keep the README skimmable: purpose before mechanism, basic path before optional detail, user content before operator reference.
