Hardening personal-memory после глубокого ревью

План одобрен — реализация идёт Последовательные волны Production не изменён и не развёрнут

Создано 2026-07-19 для репозитория personal_memory. Исходное состояние: main на a693256, рабочее дерево чистое.

1. Задача и границы

Последовательно устранить подтверждённые ревью риски потери данных, несогласованности Qdrant, деградации RAG, зависаний внешних клиентов, ошибок Todoist, неточных recall-метрик, квадратичной нагрузки viz и недетерминированных deployment-зависимостей.

Работа идёт небольшими тестируемыми волнами. Каждая волна должна оставлять сервис в рабочем состоянии и заканчиваться checkpoint’ом пользователя. Автоматический deploy, изменение production-конфигурации в personal_ai_stack, push, PR и commit не входят в разрешённые действия без отдельного явного запроса.

Не входит в scope

  • Новый UI или изменение визуального языка dashboard.
  • Переезд с Qdrant/TEI, изменение embedding-модели или размерности vectors.
  • Multi-user tenancy и новая модель OAuth scopes.
  • Восстановление фактов, уже исторически потерянных из-за одинакового text-only ID.

2. Текущее и целевое поведение

ОбластьСейчасПосле выполнения
Memory mutations update_fact/delete_fact берут ближайший result без нижнего score. Мутация отклоняется при слабом или неоднозначном совпадении; опасные числа валидируются.
Point IDs ID зависит только от текста и конфликтует между namespace. Новые ID включают namespace; legacy IDs читаются; отдельная dry-run миграция переводит существующие точки.
Qdrant writes HTTP acknowledgment принимается за завершённую запись. Критические write/delete/payload операции используют wait=true, проверяют HTTP и Qdrant status.
RAG Старые chunks удаляются до полной записи новых; пустой файл сохраняет старый индекс. Новая версия файла полностью записывается первой; после подтверждения удаляются предыдущие версии.
Reliability Qdrant/TEI без client timeout; server shutdown обрывает запросы. Ограниченные HTTP timeouts, bounded bodies, graceful shutdown и корректная обработка status codes.
Auxiliary paths Todoist filter не escaped; recall counts теряются; viz делает все pairwise comparisons. Корректные URL/schema, сериализованные increments, hard caps и bounded top-K для viz.
Build/deploy Mutable images/model revision и скачиваемые browser assets без lock/checksum. Закреплённые версии/digests, проверяемые browser assets и воспроизводимая test-команда.

3. Выбранный подход и альтернативы

Рекомендация: независимые волны с checkpoint после каждой

Сначала закрыть прямые data-loss сценарии без миграции, затем отдельно выполнить ID migration и RAG versioning, после чего заниматься доступностью, метриками, viz и supply chain. Такой порядок минимизирует blast radius и позволяет остановиться после любой волны с рабочим сервисом.

ВариантПлюсыМинусыРешение
Один большой hardening change Один общий релиз. Сложное ревью, миграция смешана с runtime-изменениями, тяжёлый rollback. Отклонён.
Исправить только P1 Быстро снижает основной риск. Оставляет подтверждённые функциональные и эксплуатационные дефекты. Подходит как промежуточная остановка после волн 1–3.
Шесть независимых волн Проверяемые границы, простой rollback, понятное продвижение step-by-step. Больше checkpoint’ов и потенциальных commit boundaries. Выбрано.

Предлагаемые commit boundaries описаны только как логические точки. Фактические commits выполняются исключительно после отдельного разрешения пользователя.

4. Архитектурные решения

Безопасные mutations

  • Добавить MUTATION_MATCH_THRESHOLD с консервативным default 0.90 и validation 0 < threshold <= 1.
  • Искать top-2 candidates; отклонять mutation, если лучший score ниже threshold или два кандидата практически неразличимы.
  • В error result возвращать candidate text/ID/score для уточнения запроса, но ничего не менять.
  • Добавить optional exact point_id для повторной гарантированной операции, сохранив query-path для совместимости.

Namespace-aware IDs и миграция

func pointID(namespace, text string) string {
    return deterministicUUID(namespace + "\x00" + text)
}
  • Новые writes сразу используют ID v2; legacy numeric и text-only IDs продолжают читаться.
  • Создать отдельную команду cmd/migrate-memory-ids; default — dry run.
  • Apply-mode: snapshot → остановить все memory writes и явно подтвердить maintenance mode → scroll с vectors → full payload/vector conflict check → upsert new ID с wait=true → verify → delete old ID.
  • Миграция идемпотентна и не запускается автоматически при старте сервера.

Versioned RAG replacement

chunkPointID(filePath, fileHash string, chunkIndex int)

1. Embed all new chunks.
2. Upsert every chunk with the new file_hash and wait=true.
3. Verify expected chunk count for the new file_hash.
4. Delete points for file_path whose file_hash differs.
5. For an empty file, delete every chunk for file_path.

Folder points получают такой же cleanup: summary удаляется, если folder исчез или больше не содержит индексируемых файлов. Ошибка cleanup не маскируется как успешный reindex.

Recall counter

Cache entries сохраняют point ID. Один server-lifetime worker сериализует increments по ID, обновляет cache-visible count и пишет абсолютное значение в Qdrant с timeout/retry. Это исключает lost updates внутри текущей single-process архитектуры; distributed counter остаётся вне scope.

10. Rollback plan

  1. Каждая волна откатывается независимо возвратом её code/config diff; не смешивать волны в один обязательный deploy.
  2. Перед ID migration создать и проверить имя Qdrant snapshot, остановить все memory writers и держать их остановленными до завершения; при ошибке восстановить collection из snapshot до возобновления writes.
  3. До завершения migration не удалять legacy support.
  4. RAG полностью восстанавливается controlled reindex из исходных документов; memory collection этим не затрагивается.
  5. При проблемах pinned deployment вернуть предыдущие image digests и model revision вместе, не по отдельности.
  6. Production rollback выполняется только по отдельному deploy runbook с health/log verification.

11. Предположения и открытые вопросы

Принятые предположения

  • Сервис остаётся single-process; recall counter не обязан быть distributed.
  • Qdrant и TEI доступны только как внутренние dependencies, но клиенты всё равно обязаны иметь timeout.
  • RAG index можно перестроить из документов; memory collection требует snapshot перед миграцией.
  • Default mutation threshold 0.90 — безопасная стартовая политика, которую можно изменить после наблюдения реальных scores.

Решения, требующие отдельного подтверждения в соответствующей волне

  • Запуск apply-mode ID migration на реальной collection.
  • Конкретные immutable TEI/Qdrant image digests и embedding model commit после сверки production.
  • Механизм trusted-proxy защиты /viz и связанные изменения в другом deployment repo.
  • Любой commit, push, PR или deploy.

12. Фактическое выполнение — все волны

  • Волна 1: конфигурация стала fail-fast; memory inputs ограничены; update_fact/delete_fact отклоняют low-score и ambiguous matches и поддерживают exact point_id.
  • Волна 2: Qdrant mutations используют wait=true и проверяют application status; новые memory IDs учитывают namespace; добавлен dry-run-by-default идемпотентный мигратор с полной payload/vector collision protection, write-before-delete и обязательным подтверждением остановленных writers для apply.
  • Волна 3: RAG chunks получили generation-aware IDs; старая полная версия сохраняется до подтверждения новой; partial runs возобновляются; empty files и stale folder summaries очищаются только после безопасного walk.
  • Волна 4: Qdrant/TEI получили timeout и bounded bodies; recall worker учитывает cache hits, сериализует increments, применяет backpressure/retry и дренируется при shutdown; HTTP, backup и RAG background lifecycle завершаются bounded graceful shutdown.
  • Волна 5: Todoist query/path encoding и MCP labels schema синхронизированы, inputs валидируются; Todoist остаётся полностью выключаемым feature gate. Graph и duplicates ограничены по nodes/results и используют bounded strongest-K heaps; PATCH tags имеет body/schema limits.
  • Волна 6: auth переведён в fail-closed режим с явным dev-only escape hatch; viz проверяет trusted proxy secret; Docker/model/browser dependencies закреплены digest/revision/checksum; docs и canonical checks синхронизированы.

Фактически изменённые области: server/config/auth lifecycle, memory/Qdrant clients и migration command, RAG indexer, backup, Todoist, viz, Docker/Compose/Makefile, environment example и project documentation.

Закрытие финального review: Memory MCP и HTTP request bodies получили проверяемые byte/count caps и строгие integer-параметры без дробного усечения. ID migration сравнивает полный нормализованный payload и vector, блокирует конфликтующую группу до writes и требует явно подтверждённый maintenance mode, исключающий конкурентные writers.

Финальные автоматические checks: go test -race ./... -count=1 прошёл для всех packages; make test подтвердил SHA-256 пяти browser assets, выполнил go vet ./..., все package tests и сборку server/indexer/migration binaries; docker build --check ., docker compose config --no-env-resolution -q и git diff --check прошли.

Известные ограничения: recall counter гарантирует отсутствие lost updates только внутри одного процесса; жёсткий kill может потерять ещё не записанные increments. Viz по-прежнему выполняет O(n²) сравнения, но только внутри валидированного hard node cap. Legacy IDs остаются до отдельно подтверждённого apply migration в maintenance mode; CLI требует -confirm-writes-stopped, однако организационное условие остановки всех внешних writers оператор проверяет самостоятельно.

Не выполнялось: commit, push, PR, production migration, изменение внешнего personal_ai_stack и deploy.

13. Детали Волны 6

  • Auth переведён в fail-closed режим: пустые API-key/OAuth запрещают startup, кроме явного development-only ALLOW_INSECURE_AUTH=true; сравнение секретов выполняется constant-time.
  • Feature gate Todoist сохранён: TODOIST_TOKEN, API_KEY, client и route требуются только при ENABLE_TODOIST=true. Добавлен отдельный положительный тест выключенного Todoist без token.
  • Для viz добавлен trusted-proxy secret: приложение проверяет X-Personal-Memory-Proxy-Secret, а repository compose настраивает Traefik на overwrite header после Authentik ForwardAuth.
  • Закреплены digest Docker base, TEI/Qdrant images и immutable embedding model revision; application image для production документирован как отдельный CI sha-*/digest pin.
  • Design-system v1.4.0 загружается из точных GitHub release assets; DS и vis bundles проверяются по build/browser-assets.sha256. make test проверяет assets, vet/tests и сборку всех трёх binaries.
  • Добавлена defensive validation для search_documents: только два известных mode и integer limit 1–100; chunk maximum ограничен 1 MiB.

Выполненные checks: make test прошёл полностью: проверены SHA-256 всех пяти browser assets, go vet ./..., все package tests и сборка server/indexer/migration binaries. Полный go test -race ./... -count=1 прошёл после финальных исправлений, как и docker build --check ., Compose config validation и git diff --check.

Deployment boundary: внешний personal_ai_stack не менялся. До production нужен отдельный review/approval diff с VIZ_PROXY_SECRET, Traefik header overwrite, immutable dependency refs и выбранным application SHA/digest; затем обязательны health/auth/log checks.
