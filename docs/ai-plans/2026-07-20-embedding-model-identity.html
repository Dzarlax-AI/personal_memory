Fail-closed embedding model identity

Реализовано локально Issue #15 · один план, без отдельной спецификации

Создано 2026-07-20. План основан на origin/main@20e06ee и read-only проверке production.

1. Цель и границы

Цель: перед обслуживанием MCP-запросов доказать, что TEI использует тот же immutable model identity, которым были созданы vectors в каждой Qdrant collection.

Архитектура: сервер получает фактический identity через TEI /info, сверяет его с закреплёнными EMBED_MODEL/EMBED_MODEL_REVISION, затем сравнивает canonical identity с metadata collections memory, doc_chunks и doc_folders. Проверка выполняется до HTTP listener, recall worker и RAG scheduler.

Стек: Go, TEI HTTP API, Qdrant REST collection metadata (Qdrant 1.16+), Docker Compose.

В scope

  • TEI /info client и strict expected-identity validation.
  • Чтение, создание и patch Qdrant collection metadata.
  • Двухфазная проверка всех активных collections до любых metadata writes.
  • Явная одноразовая adoption существующих non-empty collections.
  • Server и standalone indexer startup guard.
  • README, Compose, unit/HTTP-contract tests, controlled production rollout и rollback.

Вне scope

  • Автоматический re-embedding при mismatch.
  • Смена embedding-модели, pooling, dtype или vector dimension.
  • Изменение пользовательских point payloads или IDs.
  • Рестарт shared Qdrant/Traefik/Authentik.

4. Canonical identity и metadata contract

В metadata каждой collection хранится один namespaced ключ personal_memory.embedding:

{
  "schema_version": 1,
  "provider": "tei",
  "model_id": "intfloat/multilingual-e5-small",
  "model_revision": "614241f622f53c4eeff9890bdc4f31cfecc418b3",
  "model_dtype": "float32",
  "pooling": "mean",
  "vector_size": 384
}
  • EMBED_MODEL обязателен после defaulting и должен точно совпасть с TEI model_id.
  • EMBED_MODEL_REVISION должен быть 40-символьным hexadecimal Hub commit и точно совпасть с TEI model_sha.
  • model_dtype, pooling и фактический vector_size входят в equality contract.
  • TEI build version остаётся diagnostic field в логах, но не входит в vector-space identity: обновление runtime допустимо только если model contract неизменен.
  • Metadata не добавляется в пользовательские points и поэтому не влияет на search, scroll, export или visualization.

Qdrant официально поддерживает collection metadata начиная с 1.16; production работает на 1.17.0. Код возвращает понятную compatibility error, если metadata API недоступна.

  2. Снять counts/status и snapshots отдельно для memory, doc_chunks, doc_folders; сохранить текущие app/compose/env rollback copies.
  3. Хешировать deterministic TEI probe response; подтвердить current image digest 492506e6… и cache commit 614241f….
  4. Обновить production Compose: заменить TEI tag на уже используемый exact digest и main на уже загруженный exact commit; app пока не менять.
  5. Пересоздать только TEI, проверить /info и exact probe hash. При любом отличии вернуть старую command и остановить rollout.

  7. Проверить health, unauth/auth MCP, Viz proxy boundary, fact/document search, counts, restart count и WARN/ERROR.
  8. Удалить one-shot flag либо поставить false, пересоздать только app и убедиться, что обычная strict verification проходит.

9. Rollback

  • Вернуть предыдущий immutable app image и сохранённый Compose; metadata безопасно оставить — старый app её игнорирует.
  • Если TEI probe изменился, вернуть прежнюю command revision main и тот же cached ref; не запускать writes новым app.
  • Collection metadata update не меняет points/vectors. Snapshots нужны как дополнительная страховка, restore выполняется только при доказанном повреждении данных и отдельном разрешении.
  • Не удалять collections, volumes, snapshots и не запускать reindex автоматически при rollback.

10. Manual QA checklist

  • Exact identity: server/indexer стартуют.
  • Изменённый model revision: server не открывает порт и называет проблемную collection.
  • Missing metadata + non-empty + flag false: fail closed без metadata writes.
  • Missing metadata + non-empty + flag true: все collections получают одинаковый record.
  • Повторный старт flag false: strict match проходит.
  • Mismatch + flag true: всё равно fail closed.
  • RAG disabled: проверяется только memory.
  • Todoist disabled: identity change не регистрирует Todoist route.
  • Facts/chunks counts не меняются от adoption.
  • Direct Viz без proxy secret остаётся 401.

11. Допущения и открытые вопросы

  • Выбрано: Qdrant collection metadata, а не служебные points или отдельная metadata collection.
  • Выбрано: Hub commit строго 40 hex; mutable branch/tag не поддерживается для persistent vector collections.
  • Выбрано: one-shot adoption — единственный override и только для отсутствующей metadata; mismatch не overrideable.
  • Открытых продуктовых решений, блокирующих реализацию, нет.

13. Фактическая реализация и проверки

Что изменено

  • Добавлены strict configuration defaults и validation для immutable model ID/revision и one-shot adoption flag.
  • TEI client читает bounded typed /info; startup сверяет model ID, exact revision, dtype, pooling и размер probe vector.
  • Qdrant client поддерживает collection metadata и отдельный POST /points/count с exact=true. При legacy adoption approximate points_count не используется как safety signal.
  • Новый coordinator сначала полностью проверяет все активные collections без записей, затем создаёт или привязывает metadata и повторно читает её для подтверждения persistence.
  • Server и standalone indexer используют один guard; прежние независимые paths создания collections удалены, RAG оставляет только payload indexes.
  • Compose, example env, README и maintainer docs обновлены под новый operational contract.

Выполненные проверки

  • go test -race ./... -count=1 — PASS.

Ограничения и дальнейшие действия

  • Production пока не изменялся; issue закрывается только после merge и controlled rollout из раздела 8.
  • Первый production startup нового image намеренно потребует одноразовый adoption flag для уже заполненных collections.
  • Автоматический re-embedding остаётся вне scope: любой настоящий identity mismatch блокирует startup.
