Implementation Plan: OpenAI ChatGPT MCP OAuth

Approval required before implementation

Created 2026-05-31 for D:\Projects\Code\Personal\personal_memory.

Task Summary

Add OAuth/OIDC-compatible authentication for the existing remote MCP endpoints so the personal-memory server can be connected as a ChatGPT Apps/connector MCP server, while preserving the current static API_KEY behavior for Claude, mcp-remote, and direct Responses API use.

The OpenAI API remote MCP path can use a server_url plus optional authorization token. ChatGPT Apps authenticated MCP onboarding is stricter: it expects MCP OAuth 2.1 discovery, a protected-resource metadata endpoint, an OAuth authorization server, and bearer access tokens issued by that flow.

Current Behavior

  • cmd/server/main.go creates two Streamable HTTP MCP servers: /memory and optional /todoist.
  • internal/middleware/auth.go accepts either X-API-Key or Authorization: Bearer <API_KEY>.
  • Unauthenticated requests currently return plain 401 unauthorized without WWW-Authenticate metadata.
  • There is no /.well-known/oauth-protected-resource route.
  • There is no JWT/OIDC validation, JWKS fetching, issuer validation, audience validation, or scope validation.
  • docker-compose.yml exposes /memory, /todoist, and /health through Traefik for the repo-local compose file.
  • The production deploy in D:\Projects\Code\Personal\personal_ai_stack\deploy\memory\docker-compose.yml exposes /memory and /health, but not /todoist.
  • Authentik is already present as a Traefik ForwardAuth layer for /viz, but it is not used by application-level MCP auth.

Desired Behavior

  • Existing clients using API_KEY continue to work without config changes.
  • ChatGPT can discover protected resource metadata at https://mcp.<domain>/.well-known/oauth-protected-resource.
  • When ChatGPT calls an MCP endpoint without auth, the server returns 401 with a valid WWW-Authenticate challenge pointing to the protected-resource metadata URL.
  • When ChatGPT sends an OAuth bearer access token, the server validates issuer, audience/resource, expiration, and required scopes before invoking MCP handlers.
  • Tool metadata should include read/write/destructive hints where supported by mcp-go, so ChatGPT has better approval and safety signals.
  • Deployment documentation clearly distinguishes API-key auth from ChatGPT OAuth auth.

Assumptions and Unknowns

Assumptions

  • Use Authentik as the authorization server if it can support the required ChatGPT OAuth flow for this app.
  • Keep API key auth enabled by default; OAuth is additive and controlled by env vars.
  • Expose only the /memory MCP server in the first ChatGPT app. Todoist is currently out of scope.
  • Use one coarse MCP scope first, such as memory:mcp, unless ChatGPT/Auth provider setup makes fine-grained scopes cheap.
  • Target personal, single-user use for this implementation. Do not design a public multi-user connector in the first pass.
  • Do not implement a custom OAuth authorization server inside personal-memory unless Authentik cannot support the required flow.

Unknowns to Resolve Before Code

  • Whether Authentik supports ChatGPT's selected client registration mode for this app: predefined client, DCR, or CIMD.
  • Whether Authentik can preserve the OAuth resource parameter and issue tokens with the expected audience claim for https://mcp.<domain>.
  • The exact production ChatGPT callback URL shown in the app management page, which must be allowlisted in Authentik.
  • Whether one broad scope (memory:mcp) is sufficient for ChatGPT onboarding, or whether Authentik/OpenAI setup forces more granular scopes.

Proposed Architecture

  1. Add an OAuthConfig section to config.Config: OAUTH_ENABLED, OAUTH_ISSUER, OAUTH_RESOURCE, OAUTH_AUDIENCE, OAUTH_SCOPES, and optionally OAUTH_JWKS_URL if provider discovery is insufficient.
  2. Keep API key auth as the compatibility path. If API_KEY matches, allow the request.
  3. If API key does not match and OAuth is enabled, validate Authorization: Bearer <jwt> as an OAuth access token.
  4. If neither auth path succeeds, return a challenge: 
    WWW-Authenticate: Bearer resource_metadata="https://mcp.example.com/.well-known/oauth-protected-resource", scope="memory:mcp"
  5. Serve protected-resource metadata: 
    {
  "resource": "https://mcp.example.com",
  "authorization_servers": ["https://auth.example.com/application/o/personal-memory/"],
  "scopes_supported": ["memory:mcp"],
  "resource_documentation": "https://mcp.example.com/memory/operational"
}
  6. Use an interface for token verification so unit tests can cover middleware behavior without needing live Authentik or network access.

Auth and Permission Model

  • Compatibility: API key remains valid for existing MCP clients.
  • OAuth bearer: Valid JWT from the configured issuer is accepted only when audience/resource and scope checks pass.
  • Scopes: First pass should use memory:mcp for the whole memory endpoint. Optional later split: memory:read and memory:write.
  • Tool exposure: First ChatGPT app should expose memory/RAG only. Todoist tools remain out of scope.
  • Audience/resource: Exact Authentik behavior is unknown. Verify whether Authentik can issue access tokens with a stable audience/resource claim before implementing token validation.
  • Users: This is for personal use. Multi-user account mapping, tenant isolation, and per-user memory stores are explicitly out of scope.
  • Approvals: ChatGPT-side approvals are useful but not a substitute for server-side scope validation.

Deployment Impact

  • The public route must include /.well-known/oauth-protected-resource; otherwise ChatGPT cannot discover auth metadata.
  • Production compose currently exposes only /memory and /health for MCP. It must be updated for the well-known path.
  • New env vars must be added to production .env or compose environment without committing secrets.
  • Authentik must allow the ChatGPT callback URL from the app management page.
  • If Authentik cannot issue a resource/audience-compatible token, pause and decide between an Authentik customization and a small dedicated OAuth authorization server.

Risks and Edge Cases

  • Authentik may not support the exact MCP OAuth profile ChatGPT expects without extra configuration.
  • Audience/resource mismatch is the most likely integration failure: ChatGPT sends resource, but the provider may issue tokens with only client-id audience.
  • OIDC ID tokens are not the same as OAuth access tokens. The server should validate the token type and claims expected for API access.
  • Accepting any valid Authentik token without audience/scope checks would over-authorize unrelated clients.
  • Adding OAuth must not accidentally disable API key auth for existing clients.
  • Tool annotations improve model behavior but do not enforce permissions by themselves.
  • Production deploy changes live in a sibling repository; code and deploy changes need coordinated review.

Rollback Plan

  • Set OAUTH_ENABLED=false or remove OAuth env vars to return to API-key-only behavior.
  • Revert the production Traefik route for the well-known path if it causes routing issues.
  • Redeploy the previous GHCR image tag if the new binary breaks existing clients.
  • Keep Authentik app changes isolated so the OAuth client can be disabled without affecting /viz ForwardAuth.
