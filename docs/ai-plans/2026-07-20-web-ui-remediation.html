Web UI Remediation

Production-informed improvement plan for the self-hosted Memory Dashboard.

1. Task Summary

Improve the existing /viz dashboard using the production audit as evidence. The first release will focus on data minimization, load cost, responsive behavior, resilient states, and keyboard/screen-reader semantics while preserving the existing dark visual language and six-tab information architecture.

Definition of done: opening Overview does not scan the document index or expose complete payloads; every tab has explicit loading/error/empty behavior; the primary flows fit a 390 px viewport; interactive elements are keyboard operable; existing desktop routes and graph filtering continue to work.

2. Current Behavior

  • init.js calls loadDocuments() during every dashboard bootstrap, even when Overview is active.
  • GET /viz/api/documents scans and aggregates every document chunk. Production currently reports 78,064 chunks, 1,990 files, and 248 folders.
  • GET /viz/api/facts returns complete fact payloads. Overview, Forgotten, Timeline, and Graph share that response even when they only need summaries.
  • A rejected shared facts promise is retained, so dependent tabs cannot retry without reloading the page.
  • Graph assumes a successful JSON response and has no visible loading, empty, or failure state.
  • Treemap tiles and document folder headers are pointer-only generic elements. Graph has no non-canvas alternative.
  • The fixed-height, overflow-hidden shell and wide content produce horizontal overflow at 390 px. Timeline and Documents also clip long content.
  • Legacy malformed tag values appear directly in native select controls.
  • Duplicates and Forgotten identify cleanup candidates but do not offer a safe, reversible mutation workflow.

3. Considered Approaches

Recommended: focused remediation

Keep the current UI architecture, add summary/detail API boundaries, lazy loading, resilient states, responsive CSS, and semantic controls. Lowest migration risk and fastest path to a visibly better release.

Frontend rewrite

Replace the embedded vanilla UI with a framework application. Better component structure long term, but introduces a build pipeline and expands deployment and maintenance risk without being necessary for the current scale.

Cosmetic-only patch

Fix CSS and labels only. Fast, but leaves the highest-impact privacy and performance problems intact.

4. Desired Design and Scope

API and privacy boundary

  • Make the default facts response a summary DTO containing only fields required by overview, filters, forgotten, timeline, and graph rendering.
  • Add a single-fact detail endpoint for explicit detail-panel requests. Full payload remains hidden until a user selects a fact.
  • Return relative document paths only. Do not expose the absolute documents root or absolute file paths.
  • Keep existing tag update authorization and anti-CSRF header behavior unchanged.

Loading model

  • Load each heavy tab on first activation. Determine Documents availability through lightweight server-rendered configuration or a small metadata request rather than a full chunk scan.
  • Build document inventory from file-level metadata where supported. If the current Qdrant model cannot avoid the chunk scan safely, add a short-lived server cache with cancellation and explicit refresh behavior; do not silently serve unbounded stale data.
  • Clear failed client promises so Retry can make a fresh request.

Interaction model

  • Preserve existing tabs and visual identity. Use buttons for treemap tiles and folder expanders, real headings, associated labels, active-tab semantics, visible focus, and live status regions.
  • Add a graph status layer and an accessible result list synchronized with graph filters. Selecting a list item opens the same detail panel.
  • Replace unwieldy tag selects with searchable inputs backed by normalized display values. Normalization is presentation-safe; stored tag values are not rewritten in this release.
  • Add pagination or incremental rendering to Duplicates, Forgotten, and Documents to keep initial DOM size bounded.

Explicitly deferred

Merge/delete/keep actions are deferred until an undo or snapshot-backed recovery model and exact semantics are designed. This release may improve candidate inspection and navigation, but it will not introduce destructive controls.

6. Implementation Steps

  1. Add failing Go tests that assert summary endpoints omit full payloads and absolute paths, detail access remains explicit, and document responses remain bounded/cached.
  2. Introduce explicit summary and detail response types in the viz handler. Retain existing graph/duplicate computation limits and proxy authorization boundaries.
  3. Separate Documents availability/metadata from inventory loading; implement lazy loading and the least invasive safe server-side optimization supported by the current Qdrant data.
  4. Refactor shared client fetch state so loading, success, empty, failure, and Retry are explicit and rejected requests can recover.
  5. Update screen markup and event handling for semantic controls, focus management, Escape-close behavior, active-tab state, associated labels, and live announcements.
  6. Add bounded page/increment rendering to long lists and a filter-synchronized accessible list for Graph.
  7. Implement responsive rules for header, tabs, toolbars, graph, timeline, cards, and long paths at 390 px, 768 px, and desktop widths.
  8. Run automated tests and perform authenticated browser QA against representative empty, error, loading, normal, long-text, and mobile states.
  9. Update this plan with actual changes, checks, limitations, and follow-up recommendations.

8. Risks and Edge Cases

  • API compatibility: the viz API is internal but could have unknown consumers. Prefer new versioned/explicit routes or compatibility fields during the transition where inexpensive.
  • Detail lookup: Qdrant contains both legacy numeric IDs and newer string IDs. Reuse existing ID normalization and test both forms.
  • Documents performance: the collection is chunk-oriented. A correct cache may be safer than inventing an inaccurate file-level query. Cache invalidation must account for reindex completion.
  • Information disclosure: error text must not expose backend URLs, credentials, absolute paths, or raw Qdrant responses.
  • Accessibility: the canvas library cannot become fully semantic by attributes alone; the synchronized list is the functional alternative.
  • Large datasets: client rendering must remain bounded even if production fact or folder counts increase significantly.

9. Rollback Plan

Deliver the work as one isolated branch and PR with no database migration. If production verification fails, redeploy the previous immutable image SHA. The existing Qdrant collections and payloads are not rewritten, so rollback requires no data restoration. Any new routes remain additive until the old client path is removed within the same reviewed change.

10. Assumptions and Open Questions

  • The current Authentik ForwardAuth boundary remains the intended protection for all /viz routes.
  • The dashboard remains an embedded, dependency-free vanilla JavaScript UI; no frontend framework or new runtime CDN will be introduced.
  • Malformed tags will be normalized for display/filtering only. A persistent data cleanup migration is outside this release.
  • No blocking product questions remain because the user delegated prioritization. Destructive cleanup actions are intentionally deferred.

11. Implementation Result

  • Fact collection endpoints now return privacy-safe summaries. Complete payloads are available only through explicit single-fact detail requests.
  • Document responses expose relative paths only. Startup uses lightweight status, while full inventory is lazy, TTL-cached, cancellation-aware, and protected from concurrent refresh stampedes.
  • Facts, Graph, Documents, Duplicates, Forgotten, and Timeline now expose explicit loading, empty, failure, and retry states.
  • Long result sets use bounded initial rendering. Document file rows are created only after folder expansion and in 50-item pages.
  • Graph includes a synchronized keyboard-accessible result list and on-demand non-modal fact inspector. Async graph/detail/tag updates guard against stale-response races.
  • Tabs, labels, headings, treemap controls, folder expanders, live regions, focus behavior, and legacy-tag filter mapping were made semantic and keyboard operable.
  • Responsive rules now prevent page-level horizontal overflow at 390 px. Mobile cards stack, Graph remains scrollable, long paths wrap, and Timeline receives an internal scroll surface.
  • The Activity visualization is a full-width, UTC-aligned seven-row calendar heatmap with complete weeks and accessible daily labels.
  • PR review follow-up detached shared document scans from an individual request, synchronized fact-summary caches after tag edits, made tag search react to typing, and modernized the screen-reader-only clipping utility.

13. Known Limitations and Follow-ups

  • Merge, delete, and keep actions remain intentionally deferred until a reversible recovery model is designed.
  • Legacy malformed tags are normalized for display only; stored Qdrant payloads are not migrated.
  • The document inventory cache is time-based because reindex completion is external to the visualization handler.
  • Browser QA used deterministic representative data. Production verification should follow deployment without performing mutations.
  • A full screen-reader session and instrumented color-contrast audit remain recommended follow-ups; no WCAG compliance claim is made.
